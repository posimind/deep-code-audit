# 한국어 보고서 템플릿 (Stage 4)

`build_report.py` 가 `verified/*.json` 을 `defects/*.json` 과 id 로 조인해 렌더링하는
**결정적** 서식이다. 이 문서는 그 산출 형식의 명세이자, 필요 시 스크립트 없이 수동
작성할 때의 기준이다. **전 항목 한국어**(감사 대상 코드베이스의 언어와 무관).

설계 기준 독자는 **이 코드베이스에 낯선 검토자**다: 제목은 위치가 아니라 요지(claim),
파일 역할·도달 경로로 맥락을 주고, 용어 범례와 발견 색인 표로 진입 장벽을 낮춘다.
검증 과정 증거(반론·방어 표면)는 접이식으로 실어 본문 흐름을 막지 않으면서
"오탐 아닌가?" 하는 검토자가 근거를 확인할 수 있게 한다.

## 정렬·분할 규칙

- 확정(confirmed) 발견만 싣는다. false positive 는 집계 수치로만 노출.
- 정렬: **critical → major → minor**, 동순위는 파일 경로 → 시작 라인 순.
- **15건 이하** → 단일 파일 `감사보고서.md` (요약 + 발견 상세).
- **15건 초과** → 3분할:
  - `00_요약.md` — 감사 범위·통계·발견 색인·상세 파일 링크
  - `01_critical_major.md` — critical/major 상세
  - `02_minor.md` — minor 상세 (다수여도 격리되어 critical/major 대응을 방해하지 않음)

## 요약부 구성 (`00_요약.md` 또는 단일 파일 상단)

1. 대상 루트, run-id, 프로젝트 유형·목적, 감사 렌즈 우선순위, 그룹 수·라인 예산
2. **"이 보고서를 읽는 법" 범례** (접이식 `<details>`) — 심각도 3단계 정의(에이전트
   정의의 "Severity criteria" 절과 동일 문구 — 그 3행은 영어 본문 안에 한국어로 유지된다), 검증 절차·검증 점수의 의미, **[조건부]** 배지,
   오탐, 검증 노트 안내. 처음 보는 검토자용 — 스킬 내부 용어를 무설명으로 노출하지
   않기 위한 장치다.
3. 심각도별 확정 건수 표 + 합계, 오탐 배제 건수
4. **검증 불능 그룹**(있으면): `state.json` 의 `failed` 그룹을 `단계/그룹` 으로 명시 —
   감사 공백이므로 수동 확인 권장
5. **발견 목록** 색인 표 — `| ID | 심각도 | 분류 | 요지 | 위치 |`. 상세를 읽기 전
   전체를 조망하고 우선순위를 정하는 용도(분할 시 상세 파일 안내 문구 포함, ID 로 검색).
   셀 안의 `|`·개행은 이스케이프.
6. 미소진 힌트(있으면): 검증 안 된 의심 신호임을 명시하고 수동 확인 후보로 나열
7. 감사 범위 제외 목록(배제 디렉터리·패턴 + 크기 가드 등 개별 배제) — 범위 투명성

## 발견 상세 서식 (발견별 요소 — 데이터가 있는 항목만 렌더)

```
### N. 🔴 <claim — 무엇이 문제인가 한 줄>            ← [조건부]면 배지가 이모지 뒤에

- **위치**: `파일:라인범위` · `심볼`
- **심각도**: 🔴/🟠/🟡 …  ·  **분류**: 보안/동시성/…  ·  **검증 점수**: N/5(또는 N/2~3)  ·  **ID**: `gX-NNN`
- **파일 역할**: <헌터 coverage 의 한 줄 역할 — 낯선 독자용 파일 맥락>

> **[조건부]** 미확정(unknown) 기준: … (unknown 이 남은 confirmed 만)

**영향**
<impact — 실제로 무슨 일이 나는가, 비전문 언어 한 줄 (critical/major)>

**결함 코드** — `파일:라인범위` · `심볼`
```<언어 태그 — 확장자에서 유도, 문법 강조용>
<offending snippet>
```

**도달 경로**
`<entry_path — 진입점→결함 라인 호출 사슬. 낯선 독자가 실행 경로를 재구성하는 단서>`

**메커니즘**
<헌터 rationale — 왜 결함인가. 파일을 열지 않고도 읽히게 작성됨>

**실패 시나리오**
<verify 산출 failure_scenario — 어떤 입력/상태 → 어떤 잘못된 결과. 재현 관점 제공>

<details>
<summary><b>검증 노트</b> — 검토한 반론·확인한 방어 표면</summary>

**검토한 최강 반론과 기각 사유**  <rebuttal>
**확인한 방어 표면**              <guard_scan 목록>
**추가 확인 이력**                <appraisal 목록>

</details>

**개선 코드 샘플**
```<언어 태그>
<fix_sample>
```

**개선 방향**
<fix_direction>
```

## 필드 매핑 (스크립트 조인 규칙)

| 보고서 항목 | 출처 |
|-------------|------|
| 제목(요지) | `defects` finding `claim` (없으면 위치로 폴백) |
| 위치·심볼 | `defects` finding `location` |
| 심각도 | `verified` `severity_final` (없으면 finding `severity`) |
| 분류 | `defects` finding `category` |
| 검증 점수 | `verified` `score` / 기재된 criteria 수 |
| 파일 역할 | `defects` `coverage[].role` (결함 파일과 path 일치 시) |
| 영향 | `verified` `impact` |
| 결함 코드 | `defects` finding `snippet` (언어 태그는 파일 확장자에서) |
| 도달 경로 | `verified` `entry_path` |
| 메커니즘 | `defects` finding `rationale` |
| 실패 시나리오 | `verified` `failure_scenario` |
| 검증 노트 | `verified` `rebuttal` / `guard_scan` / `appraisal` |
| 개선 코드/방향 | `verified` `fix_sample` / `fix_direction` |
