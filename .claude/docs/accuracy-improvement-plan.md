# 정탐 정밀도 개선 계획 (환경 브리프 · 룰브릭 운영화)

deep-code-audit의 정탐률(true positive rate)을 높이기 위한 개선 작업 계획.
2026-07-07 검토 결과를 근거로 하며, 두 검토 질문 — (1) 감사 전 코드베이스 사전 조사가
도움이 되는가, (2) 적대적 검증의 점수 산출 기준을 명확히 하는 것이 도움이 되는가 —
에 대한 결론과 그 구현 방안을 담는다.

## 0. 배경과 결론

### 검토 결론 요약

| 질문 | 결론 |
|---|---|
| 사전 조사(프로젝트 성격·OS·아키텍처·테스트 경로 파악) | **부분적으로 이미 구현됨.** 프로젝트 성격 파악(Stage 1a 브리프)과 테스트 경로 차등화(`low` 분류 + minor 상한 기계 강제)는 존재한다. 실제 공백은 **실행 환경 사실(OS·아키텍처·동시성 모델·노출 모델)** 이 브리프에 없다는 점 — 이 공백이 검증 단계의 unknown 판정을 양산해 오탐·미탐 양쪽으로 샌다. → **트랙 A** |
| 검증 점수 기준 명확화 | **점수 산출은 이미 기계적으로 정의됨**(score = met 개수, 게이트·임계·시나리오를 `validate_output.py`가 기계 검증 — 산술에 재량 없음). 모호성이 실제로 남은 곳은 **각 기준을 met/unmet/unknown으로 판정하는 기준**이다. 점수 체계를 바꾸는 게 아니라 기준별 판정을 운영화하고 증거를 의무화하는 것이 정답. → **트랙 B** |

### 이미 구현되어 있어 재발명하지 않는 것

- Stage 1a 감사 브리프: `project_type` / `purpose` / `lens_priority` / `high_risk_areas`
  (README·매니페스트·진입점 인모델 정독).
- 테스트 경로 차등화: `select_targets.py`의 `low` 분류 → 헌터 프로토콜(심각도 상한
  minor) + 검증 스크립트의 기계 강등, Stage 1c 인모델 분류 검토.
- 검증 판정의 기계 검증: 게이트(unmet 1개 → false_positive) · 임계(met≥3) ·
  시나리오 필수 · score=met 개수 일치 — `validate_output.py` 일관성 규칙 1~5.

### 실제 공백 (이번 계획이 메우는 것)

| # | 공백 | 정탐에 새는 경로 | 트랙 |
|---|------|----------------|------|
| 1 | 브리프에 실행 환경 사실 없음 | 헌터가 Linux 전용 데몬에 Windows 경로 결함 보고(오탐), aarch64 약한 메모리 순서 결함 미격상(미탐), 사도(死道) 플랫폼 분기에 정독 시간 낭비. 검증자는 ②`reachable` 판정 근거가 없어 unknown 양산 → met≥3이면 환경 의존 오탐이 confirmed로 통과 | A |
| 2 | 무른 기준(②④⑤)의 met 판정에 증거 불요 | ⑤ `survives_rebuttal`은 반박을 실제 시도하지 않아도 met을 찍을 수 있음(형식적 통과) → 오탐 잔존. ④ `no_guard`는 탐색 없이 met 가능 | B-a |
| 3 | unmet 판정(게이트 즉사)에 근거 의무 없음 | unmet 오판 = 발견 즉사 = **미탐**. 현재 `reject_reason` 예시는 file:line을 인용하지만 규칙이 아님 | B-c |
| 4 | severity 앵커가 검증자 본문에 없음 | critical/major/minor 정의가 헌터 본문에만 있는데 `severity_final` 재조정 권한은 검증자에게 있음 → 심각도 판정이 에이전트 간 표류 | B-e |
| 5 | unknown 잔존 confirmed의 전제가 보고서에 안 드러남 | met≥3 + unknown 2개로 통과한 발견은 "환경 전제가 참이면 결함"인데 무조건부 confirmed처럼 보고됨 | B-d |
| 6 | 임계 기각에 unknown 해소 시도 의무 없음 (재검토에서 발견) | 감정 보강(appraisal)은 게이트·임계를 **통과한** 발견에만 의무 — met<3이면 unknown을 해소하려는 시도 없이 곧바로 기각된다. unmet 기각에는 반증이 필요하지만 unknown 기각에는 아무 근거도 필요 없는 비대칭. B-b가 met 판정 비용을 올리면 검증자가 unknown으로 도피할 유인이 생기고(unknown 인플레), 이 비대칭을 타고 진짜 결함이 임계 미달로 죽는 **미탐 채널**이 증폭됨 | B-f |
| 7 | 렌즈 우선순위의 배제적 해석 | 헌터 본문은 "lens_priority 순으로 무게를 둔다"고만 말함 — 하위 렌즈를 아예 적용하지 않는 해석이 가능하고, 가설 기반 탐지에서 가설 없는 렌즈에서는 발견이 나올 수 없음(미탐) | C-1 |
| 8 | 런 중 대상 수정(드리프트) 무방비 | 수 시간짜리 런 도중 사용자가 대상을 수정하면 헌터가 본 코드와 검증자가 보는 코드가 어긋남 — 판정 자체가 무의미해짐 | C-2 |
| 9 | 감사 대상 코드 실행에 대한 규칙 부재 | Bash가 허용돼 있어 "실행해서 확인"이 가능 — 악성 저장소의 인젝션 방어를 우회하는 실행 채널이자, 로컬 환경≠대상 환경으로 인한 reachable/harmful 오판 채널 | C-3 |

### 비목표

- **점수 체계 변경 금지**: 기준별 가중치·부분 점수(기준당 0~5 세분화)는 판단을 숫자에
  숨기는 거짓 정밀도이며, 스크립트가 검증할 수 없는 재량을 늘린다. 3상태 + 게이트 +
  임계 구조는 유지한다.
- **스크립트에 판단 이식 금지**: 환경 판정(OS/아키텍처)을 키워드 휴리스틱 스크립트로
  만들지 않는다 — 브리프와 같은 이유로 인모델 판단이다(설계 원칙 1).
- **light 룰브릭 비용 증가 금지**: minor 검증은 의도적으로 저비용이다. 새 증거 의무는
  full 룰브릭에만 적용한다(minor를 격상하면 full 재채점 의무가 이미 있으므로 자동
  적용됨).
- **새 에이전트·새 스테이지 추가 없음**: 두 트랙 모두 기존 산출물의 필드 추가 +
  에이전트 본문 절 추가 + 검증 규칙 추가로 끝난다.

### 검토했으나 보류한 항목 (재검토 기록)

| 항목 | 보류 사유 |
|---|---|
| 검증자가 재도출 중 발견한 **신규 결함의 보고 채널** (claim과 다른 실결함을 같은 위치에서 발견하는 경우) | 탐지/검증 단계 분리를 훼손하고, 그 발견을 누가 검증하느냐는 소비 루프가 추가로 필요. 탐지 측이 이미 primary+sweep+second 3중이라 기대 수율 낮음. M4 issues 데이터에서 실제로 관측되면 재평가 |
| 렌즈별 coverage 항목화(파일마다 5렌즈 확인 기록) | 설계 원칙 3(체크리스트 금지) 위반 — 체크리스트化는 목록 밖 결함을 놓치게 한다. C-1의 1줄 가중 지시로 갈음 |
| 검증자의 동적 실행(결함 재현 실행)으로 정밀도 향상 | 대상 코드 실행은 인젝션→실행 채널 + 로컬 환경 차이 오판 채널 — 오히려 C-3에서 반대로 금지를 명문화 |
| 기각된 발견의 재심(appeal) 루프 | 비용 대비 효과 불명. 2c(unmet 근거)·2f(임계 기각 전 해소 의무)가 기각 오판의 주 채널을 먼저 막는다 — M3 측정에서 기각 오판이 여전히 관측되면 재평가 |

## 1. 트랙 A — 브리프 `environment` 절 (실행 환경 사실 기록)

### 1a. 스키마 (schemas.md §1 `brief`에 추가)

```jsonc
"brief": {
  "project_type": "web_service",
  "purpose": "...",
  "lens_priority": ["security", "..."],
  "high_risk_areas": [...],
  "environment": {
    "os_targets":        {"value": ["linux"],            "evidence": ".github/workflows/ci.yml 매트릭스가 ubuntu만, Dockerfile FROM debian"},
    "arch_targets":      {"value": ["x86_64", "aarch64"], "evidence": "CI 매트릭스 + .cargo/config.toml 타깃"},
    "concurrency_model": {"value": "tokio 멀티스레드 런타임", "evidence": "main.rs #[tokio::main(flavor=multi_thread)]"},
    "runtime":           {"value": "Rust 1.75+ / Python 3.9+", "evidence": "Cargo.toml rust-version, pyproject requires-python"},
    "exposure":          {"value": "인증 없는 공용 HTTP 수신", "evidence": "README 배포 절, docker-compose 포트 공개"}
  }
}
```

- 각 항목은 `value` + `evidence` 쌍. **`value: "unknown"` 허용** — 근거를 못 찾으면
  단정하지 말고 unknown으로 남긴다(`evidence`에 "확인 시도한 위치" 기록).
- `runtime`(재검토에서 추가): 언어 런타임·버전 범위. 버전 의존 동작(표준 라이브러리
  시맨틱 변경, TLS/암호화 기본값, 정수 오버플로 처리 등)의 reachable/harmful 판정에
  쓰인다 — OS/아키텍처와 같은 성질의 환경 사실이다.
- `environment` 전체는 **선택 필드**(구 런의 groups.json과 하위 호환).
  `validate_output.py`의 §1 검사는 "존재하면 형태 검증"으로 추가한다.
- `group_by_lines.py`가 brief를 groups.json에 병합하는 기존 경로를 그대로 타므로
  스크립트 로직 변경은 없다(스키마 검사만 갱신).

### 1b. 판정 근거 소스 (SKILL.md Stage 1a에 명시)

브리프 작성 중 이미 읽는 파일들 옆에 있다 — 추가 비용은 파일 몇 개 열람 수준:

- CI 워크플로 매트릭스(`.github/workflows/`), Dockerfile / docker-compose
- Cargo.toml + `.cargo/config.toml` + build.rs / go.mod + build tags(`//go:build`) /
  pyproject classifiers / `package.json engines`
- 조건부 컴파일·플랫폼 분기 인벤토리: `#ifdef _WIN32` · `cfg(target_os)` ·
  `runtime.GOOS` grep (분기의 존재 자체가 멀티 플랫폼 신호)
- README 설치·배포 절

### 1c. 오단정 방지 3규칙 (이 설계의 핵심)

잘못된 "Linux 전용" 단정은 헌터가 Windows 경로를 통째로 안 보게 만드는 **체계적 미탐
채널**이라, 없느니만 못할 수 있다. 따라서:

1. **가설 가지치기(결함 후보 배제)는 명시 근거로 확정된 환경 사실로만** 한다. 배제에
   사용한 근거는 헌터가 rationale에 인용한다("CI 매트릭스가 ubuntu 전용이므로 Windows
   경로 처리 미보고"). evidence 없는 항목으로는 가지치기 금지.
2. **unknown 환경이면 보고하되 환경 전제를 rationale에 명시**한다("Windows에서 실행될
   경우 …"). 판정은 검증자와 최종 보고서(트랙 B-d의 조건부 표기)로 넘긴다.
3. **검증자는 ②`reachable`/③`harmful` 판정 시 brief.environment의 evidence를 직접
   재확인**할 수 있다(브리프를 맹신하지 않음 — evidence가 가리키는 파일을 연다).

### 1d. 에이전트 본문 델타

- **헌터** (`agents/deep-audit-hunter.md` 입력 절): "brief의 `environment`를 읽고
  탐지 가설의 환경 전제로 삼아라" + 위 3규칙 중 1·2 + aarch64 포함 시 약한 메모리
  순서(배리어·acquire/release 누락) 관련 동시성 렌즈 강화 1줄.
- **검증자** (`agents/deep-audit-verifier.md` 룰브릭 절): ②·③ 판정 시
  brief.environment 참조 + 규칙 3. unknown 환경 전제로 confirm할 때는
  `failure_scenario`에 전제를 명시(B-d와 연동).

### 1e. 부수 항목 — 테스트 경로 관례 대조 명문화

Stage 1c 분류 검토 항목에 1줄 추가: 브리프 작성 중 파악한 **관례 밖 테스트 경로**
(`spec/`, `e2e/`, golden 파일 디렉터리 등)가 `targets.json`에서 `low`로 분류됐는지
대조하고, 아니면 `--include`/`--exclude`로 교정. (신규 메커니즘 아님 — 기존 1c 검토의
체크 항목 명문화.)

## 2. 트랙 B — 룰브릭 기준 운영화 + 증거 의무화

### 2a. 기준별 판정 기준 표 (검증자 본문에 추가)

다섯 기준의 "무름 정도"가 다르다 — ①은 코드 대조로 비교적 객관적이지만 ②④⑤는 판정
기준이 없다. 검증자 본문의 룰브릭 절에 다음 컴팩트 표를 추가한다:

| 기준 | met 조건 | unmet 조건 | unknown 조건 |
|---|---|---|---|
| ① does_this | 주장한 동작을 코드 라인으로 확인 | 코드가 다르게 동작함을 확인 | (원칙적으로 없음 — 코드가 있으면 판정 가능) |
| ② reachable | 진입점→결함 라인의 **구체 호출 경로를 식별**(`entry_path`에 기록) | 모든 지원 구성에서 사도(dead)임을 확인(플래그 상시 off, 미호출 코드 등) | 런타임 설정·배포 구성 의존이라 저장소만으로 확정 불가 |
| ③ harmful | C/I/A 손실·데이터 오염·크래시 등 실질 피해를 심각도 앵커와 대조해 확인 | 피해가 이론상뿐임을 확인(예: 공격자가 이미 그 권한 보유, 피해 대상이 임시 데이터) | 피해 규모가 외부 시스템·데이터 특성에 의존 |
| ④ no_guard | 호출부·미들웨어·설정을 **실제 탐색하고 목록화**(`guard_scan`에 기록)한 뒤 방어 부재 확인 | 방어 위치를 file:line으로 확인 | 방어가 저장소 밖(인프라 WAF 등)에 있을 수 있어 확정 불가 |
| ⑤ survives_rebuttal | **최강 반론을 서술하고 그것이 왜 실패하는지 기록**(`rebuttal`에 기록) | 반론이 성립(발견이 죽음) | (원칙적으로 없음 — 반박 시도는 항상 가능) |

### 2b. 스키마 확장 — met 판정 증거 필드 3종 (schemas.md §3)

coverage에서 이미 쓴 패턴("경로 명단만으로는 정독 증거가 되지 않는다")을 무른 세
기준에 적용한다. §3 result 객체에 조건부 필수 필드 3종 추가:

| 필드 | 형태 | 필수 조건 |
|---|---|---|
| `entry_path` | string — 진입점→결함 라인 호출 사슬 한 줄 (`"main → route /search → handle_search:42"`) | `rubric: "full"` 이고 `criteria.reachable == "met"` |
| `guard_scan` | string[] — 실제 확인한 표면 목록 (`["app.py 미들웨어 체인", "nginx.conf", "호출부 3곳"]`) | `rubric: "full"` 이고 `criteria.no_guard == "met"` |
| `rebuttal` | string — 최강 반론 + 그것이 실패하는 이유 | `rubric: "full"` 이고 `criteria.survives_rebuttal == "met"` |

- `validate_output.py` 일관성 규칙 **6~8**로 추가: 해당 기준이 met인데 대응 필드가
  없거나 비어 있으면 스키마 불합격 → 오류 첨부 재시도(기존 재시도 사다리 그대로).
- **light 룰브릭은 면제**(비목표 참조). minor→major/critical 격상 시 full 재채점
  의무가 이미 있으므로 격상 발견에는 자동 적용된다.
- ⑤가 특히 중요하다: 현재는 반박 시도 없이 met을 찍어도 아무것도 걸리지 않는다.
  반론을 글로 쓰게 하면 형식적 통과가 구조적으로 어려워진다 — 이 필드가 트랙 B의
  오탐 감소분 대부분을 담당할 것으로 예상.

### 2c. unmet 판정 근거 의무 (게이트 오판 = 미탐 방지)

게이트 구조상 unmet 하나가 발견을 즉사시키므로 **unmet 오판은 곧 미탐**이다.
2b가 오탐 방지라면 2c는 그 대칭이다.

- 검증자 본문에 명문화: "unmet 판정은 반증 근거(file:line 또는 구성 파일 위치)를
  `reject_reason`에 인용하라. '도달 안 될 것 같음' 수준의 추정으로 unmet을 찍지 마라 —
  확정 못 하면 unknown이다."
- 기계 검증(경량): 게이트로 기각된 결과(`criteria`에 unmet 존재)의 `reject_reason`이
  **어느 기준이 unmet인지 명시**하는지 검사(기준 키 또는 ①~⑤ 표기 포함 여부).
  file:line 패턴 강제는 하지 않는다 — "플래그 상시 off" 같은 정당한 반증이 단일
  위치를 갖지 않는 경우가 있어 기계 규칙으로는 과잉이다(본문 의무 + M3 샘플 검토로
  커버).

### 2d. unknown 잔존 confirmed의 조건부 표기

met≥3 + unknown으로 통과한 confirmed는 "전제가 참이면 결함"이다. 이를 보고서에서
정직하게 드러낸다:

- 검증자 본문: unknown이 남은 채 confirm하면 `failure_scenario`에 그 전제를 명시
  ("설정 X가 기본값일 때 …", "WAF가 없는 배포에서 …"). 트랙 A의 environment가
  기록돼 있으면 이 unknown 중 상당수는 애초에 met/unmet으로 확정된다.
- `build_report.py`: `criteria`에 unknown이 남은 confirmed 발견에 **[조건부]** 배지 +
  unknown 기준 나열. (렌더링 변경만 — 판정 로직 무변경.)

### 2e. severity 앵커를 검증자 본문에 복제

critical/major/minor 정의(헌터 본문 "심각도 기준" 절과 동일 문구)를 검증자 본문에도
넣는다. `severity_final` 재조정이 헌터와 같은 앵커 위에서 이뤄지게 하는 것 — 문구가
두 파일에 복제되므로, 수정 시 양쪽 동기화 주석을 각 절에 남긴다.

### 2f. 임계 기각 전 unknown 해소 의무 (B-b의 2차 효과 상쇄 — 미탐 방지)

현행 감정 보강(appraisal)은 게이트·임계를 **통과한** critical/major의 ②·④ unknown에만
의무다(검증자 본문 "심각도 차등 검증" 절). 임계 미달(met<3, unmet 없음) 기각은 unknown
해소 시도 없이 가능하다 — unmet 기각에는 반증이 필요하지만 unknown 기각에는 아무 근거도
필요 없다는 비대칭이 있다. 2b가 met 판정 비용을 올리면 검증자가 unknown으로 도피할
유인이 생기고(unknown 인플레), 이 비대칭을 타고 진짜 결함이 임계 미달로 죽는다.
**2b를 단독 적용하면 오탐은 줄지만 미탐이 늘 수 있다 — 2f는 2b와 같은 Phase에서
반드시 함께 적용한다.**

- 검증자 본문: critical/major를 **임계 미달로 기각하기 전에**, 남은 unknown 기준
  (②·③·④)의 met/unmet 확정을 감정 보강으로 시도하고 그 이력을 `appraisal`에 남겨라.
  해소 후에도 met<3이면 그때 기각한다.
- 기계 검증(일관성 규칙 9): `rubric: "full"` · `verdict: "false_positive"` ·
  `criteria`에 unmet 없음(= 임계 기각)이면 `appraisal`이 비어 있으면 안 된다.

## 3. 트랙 C — 경량 보강 3건 (재검토에서 추가)

각 1~3줄 규모의 보강. 정탐에 직접 닿는 공백이지만 트랙 A·B의 구조에 속하지 않는 것들.

### C-1. 렌즈 우선순위는 가중이지 배제가 아니다 (헌터 미탐 방지)

헌터 본문은 "브리프의 `lens_priority` 순으로 무게를 둔다"고만 말한다 — 하위 렌즈를
아예 적용하지 않는 해석을 막는 1줄을 추가한다: "우선순위는 가중이지 **배제가 아니다** —
하위 렌즈도 각 파일 정독에서 최소 한 번은 가설을 세워본다." 탐지 절차가 가설 기반이므로
가설 없는 렌즈에서는 발견이 구조적으로 나올 수 없다 — 이 해석 차단은 순수 미탐 방지다.
(렌즈별 체크리스트化는 하지 않는다 — 보류 항목 참조.)

### C-2. 런 중 대상 드리프트 가드 (검증 무결성)

런은 수 시간 걸릴 수 있고, 사용자가 그 사이 대상을 수정하면 헌터가 본 코드와 검증자가
보는 코드가 어긋난다 — 위치·라인이 어긋난 채 검증하면 오탐·미탐 이전에 **판정 자체가
무의미**해진다.

- Stage 0b에서 `git rev-parse HEAD` + `git status --porcelain` 요약을
  `$RUN/preflight/vcs.json`에 기록(mode.json과 같은 비스키마 정보 파일 —
  오케스트레이터가 직접 쓴다).
- Stage 3 진입 전(및 재개 시) 재확인 — 달라졌으면 사용자 고지 + `log-issue` 후 진행
  여부를 확인한다. 비git 대상은 기록 생략을 명시.
- 스크립트 무변경(오케스트레이터 절차만 추가).

### C-3. 감사 대상 코드 실행 금지 명문화 (판정 오염·인젝션 실행 채널 차단)

두 에이전트 본문 어디에도 대상 코드 실행에 대한 규칙이 없다 — Bash가 허용돼 있으므로
검증자가 "실행해서 확인"을 시도할 수 있다. 이는 (a) 저장소 내용을 신뢰할 수 없는
데이터로 취급하는 인젝션 방어를 우회하는 **실행 채널**이고(코드 실행 자체가 페이로드),
(b) 감사자 로컬 환경이 대상 환경과 달라 reachable/harmful을 **오판하는 채널**이다
(트랙 A가 기록하는 environment와 로컬 환경은 다를 수 있다). 두 본문에 1줄씩 추가:
"감사 대상 저장소의 코드·테스트·빌드를 실행하지 마라 — Bash는 검색·파일 처리용이다.
판정은 정적 근거로만."

## 6. 리스크와 완화

| 리스크 | 완화 |
|---|---|
| 환경 오단정 → 체계적 미탐 (없느니만 못한 케이스) | evidence 필수 · unknown 허용 · 가지치기는 명시 근거로만(1c 3규칙). 검증자가 evidence 재확인 가능 |
| 검증자 본문 비대 → 프로토콜 희석 | 판정 기준은 컴팩트 표 1개로, 예시 나열 금지. 추가분 합계 ~25줄 목표 |

| 진행 중 런과의 스키마 비호환 | 새 필드는 전부 "선택 또는 조건부 필수"로 추가 — 구 런의 done 그룹은 재검증하지 않으므로 영향 없음. 진행 중 런 재개 시 verify 미완 그룹부터 새 규칙 적용(혼합 산출 허용 — 규칙 6~8은 신규 산출에만 걸림) |
| minor 파이프라인 비용 증가 | light 룰브릭 면제 명시(2b) |
| **2b의 2차 효과** — met 증거 의무가 unknown 도피를 유발(unknown 인플레) → 임계 미달 기각 증가 = **미탐 증가** | 2f(임계 기각 전 해소 의무 + 규칙 9)를 2b와 같은 Phase에서 분리 불가로 묶음. M3에서 unknown 비율을 별도 집계해 감시(§7) |

## 7. 측정 (M3/M4 연계)

이 계획의 항목은 전부 M3 평가 픽스처로 전후 비교 가능한 조정이다. **베이스라인은
Phase 0에서 개선 적용 전에 만든다** — 프롬프트 변경은 토글이 불가능해 사후에는 "전"
데이터를 복원할 수 없다. M3에서 측정할 것:

- **오탐률**: confirmed 중 픽스처 정답지에 없는 발견 비율 (Phase 1 전후)
- **미탐률**: 심어둔 결함 중 미보고 비율 — 특히 게이트 기각된 진짜 결함(2c 효과)과
  환경 의존 결함(트랙 A 효과)을 분리 집계
- **unknown 비율 변화**: 트랙 A 적용 전후 ②④의 unknown 판정 비율. 추가로 Phase 1
  이후 **unknown 인플레 감시** — ②④⑤의 unknown 비율이 베이스라인보다 오르는지, 오르면
  2f의 임계 기각 전 appraisal이 이를 실제로 해소하는지(해소율)를 분리 집계
- **증거 필드 실질성**: `rebuttal`/`guard_scan` 샘플 수동 검토 — 형식적 기입이
  관찰되면 본문 지시 강화(M4 튜닝 노브)

  "프롬프트 변경은 사후 A/B 불가" 전제는 git으로 완화된다 — **개선 전 스킬은 커밋
  `14d4afb` 에 보존**되어 있으므로, M3 픽스처가 준비되면
  `git worktree add <dir> 14d4afb` 체크아웃(스킬·에이전트 심링크 임시 전환)으로
  베이스라인을 먼저 측정한 뒤 개선판을 측정하면 §7의 전후 비교가 그대로 성립한다.
  단 이 경로가 유효하려면 이번 변경을 **이력 보존 커밋으로 남겨야 한다**(rebase/amend 로
  14d4afb 를 지우지 말 것).
