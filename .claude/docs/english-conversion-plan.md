# 지시층 영어 전환 구현 플랜

deep-code-audit의 **지시층**(SKILL.md 본문, 에이전트 본문 2종, 태스크 골격, 스키마 산문)을
영어로 전환해 런당·세션당 토큰 비용을 절감하는 작업 계획. 2026-07-07 검토 결과를 근거로
하며, 산출 수치와 가정은 부록 A에 정리했다. 같은 날 "기능 저하 금지" 관점 재검토로
보강: 브리프 언어(§1), worktree 작업(Phase 0), 기계 계약 보존 검사·A/B 비교 런(§3),
착수 순서 결정 포인트(§3 말미).

## 0. 배경과 목표

### 왜 하는가

Claude 토크나이저에서 한글은 음절당 ~1.2토큰, 영어는 자당 ~0.25토큰이 나온다 — 같은
내용 기준 한국어 문서가 약 2배 토큰을 소모한다. 그리고 이 스킬의 지시층은 **곱셈
구조로 과금**된다:

| 지시층 파일 | 실리는 곳 | 곱셈 계수 |
|---|---|---|
| 에이전트 본문 2종 | 매 스폰의 시스템 프롬프트로 전 턴 상주 | 스폰 수(~22) × 턴 수(25~40) |
| `references/schemas.md` | 전 서브에이전트가 Read 후 잔여 턴 상주 | 스폰 수 × 잔여 턴 수 |
| SKILL.md 본문 | 오케스트레이터 컨텍스트 전 턴 상주 | 오케스트레이터 턴 수(~120) |
| `hunt/verify-task.md` 골격 | 매 스폰의 태스크 프롬프트 | 스폰 수 |
| frontmatter `description` 3종 | **스킬을 안 쓰는 세션 포함** 매 세션 스킬/에이전트 목록 | 전 세션 상시 |

8그룹·고위험 3그룹 규모 런 기준 지시층 유효 입력 **약 940K → 600K 토큰(−36%,
런당 ~340K 절감)**으로 추정한다(부록 A). 전체 런 비용 대비로는 4~6% 수준이다 —
감사 대상 코드 열람이 지배적이므로 과대 기대하지 않는다. 대신 절감이 **매 런·매 세션
반복**되고, 부수적으로 영어 지시의 준수율이 한국어와 같거나 낫다는 이점이 있다.

### 비목표

- **발견 산문 데이터의 영어화** — 보고서는 `build_report.py`의 결정적 렌더라 번역
  단계가 없다. 산출 JSON의 산문 필드가 한국어인 것이 한국어 보고서의 유일 경로이며,
  이는 사용자 전제이기도 하다. "영문 작성 → 취합 후 번역" 대안은 2026-07-07 정량
  평가 후 기각했다(**부록 B** — 절감이 지시층 전환의 1/3 이하인데 기계 검증 불가한
  조용한 저하 채널을 신설).
- **스크립트 4종의 주석·메시지·테스트 영어화** — 런타임 컨텍스트에 실리지 않아 절감이
  0에 가깝고 `test_scripts.py` 갱신 부담만 생긴다. (Phase 4의 경고 추가는 번역이 아닌
  기능 추가로 별개.)
- **`report-format.md`** — 한국어 보고서 템플릿의 명세 자체인 데다 런타임 로드가 없다.
- **설계 문서(`.claude/docs/`)** — 개발용, 런타임 무관, 사용자 작업 언어 유지.
- **프로토콜 의미 변경** — 이번 작업은 번역이다. 문장 강화·약화·재구성은 회귀 채널이므로
  금지. 단 **언어 정책 지시 신설(§1)은 예외** — 전환의 필수 동반 조치다.

## 1. 언어 정책 (전환 후 불변식)

단일 규칙: **모델에게 주는 지시는 영어, 사람(사용자·보고서 독자)에게 닿는 텍스트는 한국어.**

| 계층 | 언어 | 대상 |
|---|---|---|
| 지시층 | 영어 | SKILL.md 본문·description, `agents/*.md` 본문·description, `hunt/verify-task.md`, `schemas.md` 산문 |
| 데이터층 | **한국어** | 산출 JSON의 모든 산문 값 필드(아래 목록). 예외: `issues` — 개발자용, 언어 자유 |
| 렌더층 | **한국어** | `build_report.py` 리터럴, `report-format.md`, 심각도 기준 3행 |

**데이터층 한국어 필수 필드** (보고서에 그대로 실리거나 한국어 필드와 대조됨):

- 헌터(§2): `claim`, `rationale`, `coverage[].role`, `coverage[].top_risk`
  ("특이점 없음" 관례 포함 — 기계 검사 아님, 관례 유지), `cross_refs[].hint`
  (residue로 보고서 표면화)
- 검증자(§3): `rederivation`, `failure_scenario`, `impact`, `reject_reason`,
  `rebuttal`, `guard_scan[]`, `appraisal[].item/.evidence`, `fix_direction`
- **오케스트레이터(§1 brief)**: `purpose` — `build_report.py:318` 이 요약부에 그대로
  렌더한다(`**프로젝트**: {project_type} — {purpose}`). `project_type` 은 영문 라벨
  관례 유지(현 스키마 예시도 영문). `high_risk_areas[].reason`·`environment[].evidence`
  는 헌터·검증자만 읽으므로 언어 자유(한국어 권장).
- 언어 중립(코드·기호): `snippet`, `entry_path`, `location.symbol`, `fix_sample`
  (단 fix_sample 안의 주석은 한국어)

`rederivation`까지 한국어로 강제하는 이유: 편승 징후 검사가 `rederivation` ↔ 헌터
`rationale`의 **자구 수준 유사도**를 본다. 언어가 갈리면 번역 편승이 항상 "유사하지
않음"으로 판정되어 검사가 무력화된다.

**영어 파일 안에 의도적으로 남기는 한국어 블록** (잔여 한글 검사의 허용 목록):

1. **심각도 기준 3행** — hunter/verifier 본문과 `build_report.py render_legend` 3곳
   자구 동기화 불변식. 범례가 한국어 보고서에 실리는 이상 세 곳 모두 한국어 원문이어야
   불변식이 문자 그대로 유지된다(번역하면 "동일 문구" 검증이 불가능해짐).
2. **SKILL.md description의 한국어 트리거 문구** — 사용자가 한국어로 요청하므로
   자구 매칭 신호를 보존한다("코드 감사", "취약점/버그/결함 찾아줘" 등).
3. **schemas.md의 예시 값** — 산출 언어의 few-shot 역할. 규칙 산문은 영어, 예시 JSON의
   산문 값은 한국어 유지.
4. **SKILL.md가 인용하는 스크립트 메시지 리터럴** — 스크립트가 한국어로 남으므로
   `[validate:WARN] … 자기중복 의심 쌍` 같은 인용은 원문 그대로(오케스트레이터가
   stderr와 자구 대조할 수 있어야 한다).
5. **언어 정책 지시문 안의 한국어 예시** — "특이점 없음" 리터럴 등.

## 2. 단계별 작업

### Phase 0 — 준비

- 작업 브랜치 생성(`english-conversion`), 파일 단위 커밋으로 Phase별 롤백 가능하게.
- **git worktree 에서 작업한다**: `~/.claude/skills/deep-code-audit` 와
  `~/.claude/agents/deep-code-audit` 가 이 레포 본 체크아웃을 가리키는 심링크라서,
  본 체크아웃에서 브랜치를 바꾸는 순간 **모든 세션의 라이브 스킬이 번역 중간본으로
  바뀐다**. 번역은 worktree 에서 하고, 본 체크아웃 전환은 카나리/A/B 런 시점에만
  의도적으로 수행한다. 전환 작업 중 진행 중인 감사 런이 없어야 한다(런 도중 프로토콜
  교체 = 그룹 간 결과 비교 가능성 파손 — SKILL.md 모드 일관성 원칙과 동일 논리).
- 전환 전 버전에 git 태그(예: `pre-english`) — §3 A/B 비교 런과 롤백의 기준점.
- 베이스라인 확인: `python3 scripts/test_scripts.py` 63개 green,
  `python3 -m py_compile scripts/*.py`.

### Phase 1 — 에이전트 본문 2종 (최대 효과: −43%/−41%)

`agents/deep-audit-hunter.md`, `agents/deep-audit-verifier.md`:

- 본문 전체 영어 번역. frontmatter `name`·`tools` 불변, `description` 영어화
  ("다른 작업에 자동 위임 금지" 의미 보존).
- **심각도 기준 3행은 한국어 원문 유지** + 동기화 주석을 "3곳 자구 동기화 + 이 3행이
  한국어인 이유(render_legend가 보고서에 렌더)"로 갱신.
- **산출 절에 언어 정책 지시 신설(양쪽)**: §1 데이터층 필드 목록을 명시하고
  "Korean report is rendered verbatim from these fields — write them in Korean"
  로 이유를 함께 적는다. 헌터에는 coverage의 `"특이점 없음"` 리터럴 유지도 명시.
  현재는 프로토콜 전체가 한국어라 출력 언어가 암묵 보장되는데, 영어화하면 이 결합이
  끊긴다 — **명시 지시 없는 전환은 보고서에 영어 산문이 섞이는 조용한 저하 채널 신설**이다.
- 번역 원칙: 문단 단위 대응 번역(원문 diff 대조 용이), 강조·표·절 구조 유지, 조건
  문장(게이트·임계·격상 의무·2단계 열람·"unmet 확정 시 반증 인용" 등)은 의미 등가를
  문장 단위로 확인.

### Phase 2 — references/

- `schemas.md`: 규칙 산문 → 영어. **예시 JSON의 산문 값은 한국어 유지**. 문서 상단에
  Language policy 절을 신설해 §2·§3 산문 필드의 한국어 규칙을 명문화(에이전트 본문
  지시의 스키마 측 대응물).
- `hunt-task.md` / `verify-task.md`: 전체 영어(오케스트레이터용 안내 + 모드 절 블록 +
  2턴 분리 절). `{{CLAIMS_EMBED}}`는 한국어 데이터가 그대로 들어간다 — 골격이 영어라도
  무방.

### Phase 3 — SKILL.md

- 본문 영어 번역. **"사용자 대화·중간 보고·최종 요약은 한국어" 지시 1줄 신설**
  (지시층이 영어가 되면 오케스트레이터의 대화 언어가 영어로 드리프트할 수 있다).
- Stage 1a(감사 브리프) 절에 **`purpose` 는 한국어로 작성** 지시 추가 — §1 데이터층의
  요약부 렌더 경로. 대화 언어 지시만으로는 파일 산출의 언어가 보장되지 않는다.
- `description`: 영어 전환 + 한국어 트리거 문구 인용 보존. 전환 후 트리거 정확도가
  의심되면 skill-creator의 description 최적화 루프로 측정(선택, 후속).
- 스크립트 메시지 인용 리터럴은 원문 유지(§1 허용 목록 4).

### Phase 4 — 산출 언어 가드: `validate_output.py` 한글 비율 경고 (포함 확정, 2026-07-07)

- `validate --stage hunt/verify` 시 **고신호 산문 필드만**(`claim`·`rationale` /
  `rederivation`·`failure_scenario`·`impact`) 대상으로 한글이 전무한 발견 항목을
  **경고**(issues.jsonl + stderr)로 표면화. `guard_scan`·`entry_path`·`appraisal.evidence`
  는 정당한 순수 경로/기호 목록일 수 있어 검사에서 제외한다(오경고 소음은 경고 무시를
  학습시키는 채널). 불합격이 아닌 경고인 이유: snippet 인용이나 영문 식별자 위주
  claim처럼 정당한 저한글 산문이 존재한다 — 자기중복 경고와 같은 패턴("기계는 표면화만,
  판단은 모델")을 따른다.
- 페어 테스트 추가. **이 Phase만 스크립트 변경을 포함**하며 독립 커밋으로 분리한다.
- 이 스킬의 원칙(EXIT=0 조용한 저하를 기계로 표면화)과 일치 — 전환이 신설하는 드리프트
  채널(Phase 1 참조)의 상시 검출기다.

### Phase 5 — 문서 동기화

- `CLAUDE.md`: 언어 정책 요약 반영(지시층 영어 / 데이터·렌더층 한국어 / 허용 블록).
- `deep-code-audit-design.md`: 결정 기록 — 근거 수치(부록 A), 한국어 유지 목록, 언어
  정책 불변식.

### Phase 6 — 검증 (아래 §3)

## 3. 검증 계획

번역 회귀를 잡을 자동 수단(M3 평가 픽스처)이 아직 없으므로, 결정적 검사 + 카나리
필드런으로 대체한다.

1. **결정적**: `py_compile` 전체, `test_scripts.py`(Phase 4 신규 테스트 포함).
2. **잔여 한글 검사**: 전환 대상 5파일에 `grep -n '[가-힣]'` → 남은 한글이 §1 허용
   목록(심각도 3행, 트리거 문구, 예시 값, 인용 리터럴, 지시문 내 한국어 예시)에만
   해당하는지 대조.
3. **번역 충실도 리뷰**: 원문(git 이력) 대비 문단 단위 대조. 특히 조건·의무 문장의
   의미 등가 확인 —
   - 에이전트 본문: 게이트/임계/시나리오 3규칙, 격상 시 full 재채점, unmet 기각 반증
     인용, 임계 기각 전 unknown 해소, 2단계 열람 순서, 인젝션 방어(자동 주입 채널 포함),
     비대칭 기록, 읽기/보고 분리, **감사 대상 코드 실행 금지**, seam_hints 한계(정적
     절단 간선만 — 부재 ≠ 무관), 중복 보고 처리, 환경 전제 가지치기 조건(evidence
     확정 항목만)
   - 태스크 골격: 모드 절 3종의 조사 범위·미열람 규칙·ID 접두사 지정
   - SKILL.md: 재시도 사다리 3양식 구분, 인프라 실패 비강등·중단, 재개 모드 일관성,
     응집 저하·드리프트 고지 의무, 프리플라이트 실패 정책(기본 중단·호환 모드 명시 동의)
4. **기계 계약 보존 검사** — 스크립트가 산출물에 강제하는 계약을 번역문이 계속 정확히
   지시하는지 목록 대조. 이 계약들의 위반은 시끄럽게 실패(재시도 비용)하지만, 지시가
   모호해지면 재시도 루프 낭비가 상시화된다:
   - 발견 ID 형식 `g<gid>-NNN` / `-wNNN`(sweep) / `-sNNN`(second) — merge 가 ID 충돌 시
     FAIL("접두사 규약 위반")
   - coverage 에 전 core 파일 등장(미커버 시 목록 첨부 재시도), low 파일 심각도 상한
     (기계 강등), `cross_refs[].category` 필수
   - `criteria` 키 5종·enum 값(`met|unmet|unknown`, `confirmed|false_positive` — 이미
     영문, 번역 무풍지대임을 확인만), `score` = met 개수
   - met 증거 필드 조건 3종(`entry_path`/`guard_scan`/`rebuttal`) + 규칙 9(`appraisal`)
   - **게이트 기각 `reject_reason` 은 unmet 기준의 키 이름 또는 ①~⑤ 표기를 포함해야
     통과**(`validate_output.py` CRITERION_MARK, 286행 부근) — 영어 본문은 기준 **키
     이름 인용**을 지시하고, 룰브릭 표의 ①~⑤ 기호는 언어 중립이므로 그대로 유지한다
   - schemas.md **절 번호(§0~§6) 유지** — 본문·태스크 골격이 "§2/§3 을 따르라"로 참조
5. **A/B 비교 필드런** (카나리를 격상 — "기능 저하 금지" 요구의 직접 검증): 동일한
   소규모 저장소에 대해 전환 전(`pre-english` 태그 체크아웃) 1회 + 전환 후 1회를 돌려
   대조한다:
   - confirmed 발견 집합의 교집합/차집합(위치 중첩 + 동일 category 를 동일 발견으로
     매칭), 심각도 분포, 오탐 기각 사유 분포(게이트/임계/중복)
   - coverage 항목의 질(역할 요약·가설이 형식적으로 퇴화하지 않았는지)
   - 기존 체크리스트: 한국어 프롬프트 트리거, 산문 필드 한국어("특이점 없음" 관례 포함),
     rederivation 독립성, 보고서 렌더(영어 혼입 없음, 범례·색인·[조건부] 배지),
     오케스트레이터 보고 한국어, Phase 4 경고 무오탐
   - **한계를 인지하고 읽는다**: LLM 런은 비결정적이라 단회 차이 = 회귀 단정 불가.
     차이가 큰 그룹만 재런해 재현성을 확인하고, 재현되는 차이만 회귀로 취급한다.
6. 실패 시 해당 Phase 커밋 단위로 롤백. 원문은 git 이력에 남아 재번역 비용이 낮다.

### 착수 순서 — 결정됨 (2026-07-07)

**옵션 B 채택: 계획 확정 즉시 전환 + A/B 비교 런** (사용자 결정). 정량 게이트의
정석인 옵션 A(M3 선행)는 채택하지 않되, **M3 픽스처가 완성되면 영어판을 픽스처로
사후 정량 재평가**한다 — A/B 런은 착수 게이트, 픽스처는 사후 회귀 확인으로 역할을
나눈다.

**회귀 판정 기준(§3-5의 대조 결과를 읽는 규칙)**: 재런으로 **재현되는
critical/major 발견 소실**만 회귀(= 병합 차단, 해당 절 재번역)로 취급한다. minor
수준의 변동, 오탐 기각 증가(검증이 더 깐깐해진 방향), 표현 차이는 허용한다 — LLM
런의 자연 변동폭 안에서 모든 차이를 회귀로 읽으면 착수가 불가능하다.

**잔여 입력 1건**: A/B 런 대상 저장소는 Phase 6 전까지 지정 필요(전례: 2026-07-02
ssam 필드런 — 재사용 후보).

## 부록 B — 대안 평가: 영문 작성 → 최종 번역 (기각, 2026-07-07)

질문: 발견 산문을 영어로 쓰고, 취합(merge·verify)까지 영어로 진행한 뒤 보고서 직전에
확정 발견만 한국어로 번역하면 더 싸지 않은가? — 채택안(한국어로 작성해 그대로 취합,
번역 단계 없음)과 정량 비교했다.

**절감 측 추정** (부록 A와 같은 8그룹 예시: 발견 80건 중 확정 40건, core 280파일,
발견당 산문량 가정 명시. ±50% 오차로 읽을 것):

| 항목 | 한국어 일관(채택) | 영문 작성 + 최종 번역 |
|---|---|---|
| 산문 출력 토큰(헌터 findings·coverage + 검증자 결과) | ~78K | ~39K |
| 번역 스테이지 | 없음 | 입력 ~18K + 한국어 출력 ~28K |
| 검증자 입력측(defects 열람·claims 임베드, 유효 토큰) | ~115K | ~58K |
| **순절감(입력 환산, 출력 단가 = 입력 5배)** | 기준 | **~90K/런 (50~150K)** |

번역 대상이 확정 발견의 보고서행 필드뿐이라, 기각된 발견·coverage `top_risk`·
`rederivation` 등 보고서에 안 실리는 산문의 한국어 출력 프리미엄을 아끼는 구조다.
**그런데도 순절감이 지시층 전환(~340K/런)의 1/3 이하다.**

**기각 사유 — 비용이 아니라 신뢰성이 결정적**:

1. **번역 충실도는 기계 검증이 불가능하다.** ID·구조 보존은 스크립트가 확인해도 의미
   보존은 못 한다. 부정어 하나가 소실된 rationale, 완화된 failure_scenario 가
   EXIT=0·스키마 통과로 보고서에 실린다 — 이 스킬이 크래시보다 위험하게 취급하는
   실패 양식(조용한 품질 저하)을 **보고서 직전, 가장 늦은 지점에** 신설하는 셈이다.
2. **검증-보고 동일성이 깨진다.** 보고서에 실리는 문장이 검증자가 확인한 그 문장이
   아니라 번역본이 된다. 적대적 검증의 보증이 번역을 통과하며 희석되는데, 번역에는
   대응하는 검증자가 없다(헌터-검증자 같은 적대 쌍 부재).
3. **LLM 재작성 금지 원칙과 충돌.** findings 파일의 LLM read-modify-rewrite 는 이
   스킬이 금지한 소실 채널인데, 번역은 구조적으로 그 행위다. 별도 파일 + 병합·상위집합
   검사로 완화해도 1·2는 남는다.
4. **영구 복잡도.** 신규 스테이지 = 스키마 절·merge kind·state 단계·재개 로직·
   build_report 조인 변경 + 직렬 지연 1단계. ~90K/런 절감으로 상환하기엔 비싸다.

**부분 하이브리드도 검토 후 기각**: 보고서에 안 실리는 필드만 영문화(`coverage[].top_risk`,
`reject_reason` — `role` 은 어느 파일에 확정 발견이 생길지 예측 불가라 전부 한국어
필요) 시 절감 ~50K 입력 환산. 그러나 필드별 언어 매트릭스는 드리프트와 Phase 4
오경고의 소음원이고, `rederivation` 은 편승 검사(자구 유사도) 탓에 `rationale` 과
같은 언어여야 한다. 소폭 절감 대비 규칙 복잡도가 나쁘다.
