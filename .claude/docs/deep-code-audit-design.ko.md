# deep-code-audit — 구조와 동작

> 이 스킬을 처음 접하는 사람을 위한 **단일 기준 문서**다. 스킬이 추구하는 목적과
> 방향, 파이프라인의 구조와 동작, 신뢰성 장치를 설명한다. 선행 설계·계획 문서 전부
> (워크플로우 정합성 검토, 개발 계획, 파서 개선 계획, 에이전트 번들링 플랜, 영어 전환
> 플랜, 정탐 정밀도 개선 계획)를 통합·대체하며(§8 이력), 서술은 **현재 구현 상태**
> 기준이다.
>
> English version: [deep-code-audit-design.md](deep-code-audit-design.md)

## 1. 무엇을 하는 스킬인가

**deep-code-audit**은 워크스페이스(또는 지정 저장소) 전체를 다중 에이전트로 깊이
감사하여 **보안·동시성·장애 처리·로직·리소스** 결함을 찾아 **한국어 보고서**로 내는
Claude Code 스킬이다. 구현물은 플러그인 구조다:

- [.claude-plugin/plugin.json](../../.claude-plugin/plugin.json) — 매니페스트 (+자기 등재 marketplace.json)
- [skills/deep-code-audit/SKILL.md](../../skills/deep-code-audit/SKILL.md) — 오케스트레이터
- [references/](../../skills/deep-code-audit/references/) — 태스크 프롬프트 골격·스키마·보고서 서식
- [scripts/](../../skills/deep-code-audit/scripts/) — Python 스크립트 4종 + 테스트 70종 (표준 라이브러리만)
- [agents/](../../agents/) — 전용 서브에이전트 2종 (헌터·검증자)

핵심 아이디어는 한 문장이다: **관대한 적대적 탐지 → 엄격한 적대적 검증의 2단 구조로,
미탐(false negative)과 오탐(false positive)을 동시에 최소화한다.** 헌터는 낮은
확신에도 후보를 기록하고(미탐 방지), 새 컨텍스트의 검증자가 그 주장을 무너뜨리려
시도해 살아남은 것만 보고서에 싣는다(오탐 방지).

심각도는 3단계: **critical**(정상 사용에서 악용·데이터 손실·크래시) /
**major**(특정 조건에서 심각한 오작동·데이터 오염·보안 약화) /
**minor**(국소적 품질·견고성 결함). 비용과 정밀도는 critical/major에 집중한다.
Fable 모델의 장문 컨텍스트 추론·가설 기반 탐지·적대적 자기반박 능력을 전제로
설계했다(서브에이전트 모델은 하드코딩하지 않고 상속).

## 2. 설계 원칙 — 추구하는 방향

1. **판단은 모델에게, 결정적 작업만 스크립트에게.** 프로젝트 분류·결함 채점 같은
   판단에 키워드 휴리스틱을 쓰면 오분류가 미탐으로 직결된다. 스크립트는 라인 수 계산,
   그룹 분할, 스키마 검증, 병합, 보고서 렌더링만 담당한다.
2. **큰 그룹, 적은 에이전트.** 그룹 경계(seam)는 cross-file 결함 미탐의 주요 원인이다.
   그룹 라인 예산을 크게(기본 10,000라인) 잡아 경계 수 자체를 줄인다.
3. **가설 기반 탐지, 체크리스트 금지.** 패턴 목록 대조는 목록 밖 결함을 놓친다.
   신뢰 경계·데이터 흐름·상태 전이를 추적하며 "여기서 무엇이 깨질 수 있는가"를
   가설로 세우고 코드로 확인한다. (사용자가 명시적으로 "이건 꼭 봐줘"라고 한 것은
   이 원칙의 예외가 아니다: 브리프의 렌즈·전제(`lens_priority`·`high_risk_areas`·
   `environment`)로 들어가 가설을 자극하고, 헌터는 그 너머까지 계속 본다. 헌터가
   출발점으로 삼는 씨앗은 이 원칙이 배제하는 종결 체크리스트가 아니다.)
4. **탐지는 관대하게, 검증은 적대적으로 — 단 비대칭으로.** critical/major 후보는
   낮은 확신에도 기록하고, minor는 명백한 경우만 기록한다.
5. **단계 간 인계는 반드시 파일로.** 모든 산출물은 고정 스키마 JSON으로 디스크에
   기록된다. 부수 효과로 단계·그룹 단위 재개(resume)가 가능하다.
6. **조용한 실패를 구조적으로 차단한다.** EXIT=0·스키마 유효인데 품질이 무너진 상태
   (부분 커버리지, 무경고 그룹핑 폴백, 병합 중 발견 소실)가 크래시보다 흔하고
   위험하다. 각 실패 양식을 명시적으로 감지·고지한다(§5). 이 원칙은 실전 감사에서
   Rust 파서 부재가 무경고로 지나간 사건(§8-3)에서 원칙 수준으로 승격됐다.

## 3. 파이프라인 개요

오케스트레이터(메인 에이전트)가 4단계(+보강 패스 2.5)를 돌리고, 탐지·검증은 그룹별
서브에이전트에 백그라운드 병렬(동시 4~6)로 위임한다:

```
[0] 프리플라이트   전용 에이전트 정의 존재 + 세션 인식 확인 → 카나리 스폰
                   → 실패 시 기본 중단 (호환 모드는 명시 동의 시에만)
[1] 선별·그룹핑    감사 브리프(인모델) + core/low/exclude 3분류 + 분류 검토(인모델)
                   + 라인 수·import 응집 그룹핑                  ──▶ groups.json
[2] 적대적 탐지    그룹별 헌터 (전역 읽기 / 그룹 한정 보고,
                   비대칭 기록, 파일별 coverage 증거)             ──▶ defects/<gid>.json
[2.5] 보강 패스    고위험 그룹 critical 전용 2차 헌터 → cross_refs
                   힌트 라우팅 → 표적 sweep → 잔여 힌트 검사      ──▶ defects 갱신 + residue
[3] 적대적 검증    새 컨텍스트 검증자 (anti-anchoring 독립 재도출,
                   3상태 룰브릭 + 게이트, 기계 일관성 검증)        ──▶ verified/<gid>.json
[4] 보고서 생성    build_report.py — 한국어, critical→major→minor,
                   15건 초과 시 3분할                             ──▶ 감사보고서.md
```

모든 산출물은 `<감사대상 루트>/.deep-code-audit/<run-id>/`(run-id =
`YYYYMMDD-HHMMSS`)에 기록된다. **재개의 단일 근거는 `state.json`**이며, 완료 기준은
파일 존재가 아니라 **스키마 검증 통과 기록**이다(크래시로 반쯤 쓰인 산출물과 완료를
구분). 운용 문제는 `issues.jsonl`에 증상/맥락/조치/결과로 기록한다 — 스킬 개선(M4
회고)의 근거 자료다. 파일 스키마 전체 명세는
[references/schemas.md](../../skills/deep-code-audit/references/schemas.md).

## 4. 단계별 동작

### Stage 0 — 프리플라이트 (전용 에이전트 게이트)

헌터·검증자의 불변 프로토콜(읽기/보고 분리·인젝션 방어·룰브릭·anti-anchoring)은
전용 에이전트 정의([agents/](../../agents/))의 **본문(=시스템 프롬프트)**에 있고
하네스가 디스크에서 직접 로드한다. 이 배치가 주는 것: (1) 긴 런의 컴팩션이
프로토콜을 의역·축약할 수 없음, (2) 스폰마다 ~130줄 템플릿이 오케스트레이터
컨텍스트에 누적되지 않음(스폰 프롬프트 = 런 변수 몇 개), (3) `subagent_type` 명시
지정으로 에이전트 선택이 결정적.

대신 **에이전트 인식 실패 = 프로토콜 미전달**이 되므로 프리플라이트가 하드 게이트다:
정의 파일 존재 확인 → 세션 인식 확인(플러그인 스코프명/무스코프명 어느 쪽이든, 세션이
나열한 식별자를 `mode.json`에 확정 기록) → 두 타입 카나리 스폰. 실패 시 **기본
중단**이다 — state.json 재개 덕에 중단 비용이 낮고 저품질 완주가 더 비싸다.
**호환 모드**(에이전트 본문 + task 골격을 범용 서브에이전트에 이어붙임)는 저하 내용을
고지받은 사용자가 명시 동의한 경우에만 실행되고, 기록되며, 최종 보고서에 명시된다.

### Stage 1 — 감사 브리프 · 대상 선별 · 그룹핑

**브리프(인모델).** 오케스트레이터가 README·매니페스트·진입점을 직접 읽고 프로젝트
유형·목적, **렌즈 우선순위**(5개 결함 축 중 이 프로젝트에서 치명적인 것), **고위험
영역**(실행 환경 사실 포함 — OS·아키텍처·동시성 모델·노출 모델; 검증 단계 unknown
판정을 줄이는 근거)을 `brief.json`에 기록한다.

**선별(`select_targets.py`).** 파일 3분류 — `core`(일반 소스 + 텍스트 설정·인프라
파일, 최우선 스캔) / `low`(테스트·픽스처 — 스캔하되 **심각도 상한 minor**, 기계
강제) / `exclude`(벤더·빌드 산출물·생성 코드·락/데이터/바이너리 + `.deep-code-audit/`
**자기 배제** — 배제하지 않으면 2회차 실행부터 이전 보고서가 감사 대상이 되는 3중
오염이 난다). 분류 결과는 오케스트레이터가 인모델로 검토하고 `--include`/`--exclude`
오버라이드로 교정한다 — `low` 오분류는 critical을 minor로 기계 강등시키고 `exclude`
오분류는 스캔 누락이 되기 때문이다.

**그룹핑(`group_by_lines.py build`).** 라인 수 균형 + **import 그래프 연결 요소**
응집. 예산 초과 클러스터는 절단되는 import 간선 수를 최소화해 분할하고, 절단 간선은
양쪽 그룹의 `seam_hints`로 기록해 헌터에게 "이 인터페이스 경유 흐름을 우선 추적하라"
로 주입한다 — **경계를 없앨 수 없으면 경계를 조명한다.** import 파서는 best-effort
(Python, JS/TS, C/C++, Rust, Go, Java/Kotlin); 파서 없는 언어는 디렉터리 응집으로
폴백하되 `cohesion`/`unparsed_source_exts` 필드와 경고로 **저하를 반드시 드러낸다**
(원칙 6). 오케스트레이터는 저하 감지 시 기록·고지 후 진행 여부를 확인한다.

### Stage 2 — 적대적 탐지 (그룹별 헌터)

그룹당 헌터 1개를 병렬 위임한다(`subagent_type: deep-audit-hunter`, 태스크 프롬프트는
[hunt-task.md](../../skills/deep-code-audit/references/hunt-task.md) 변수 골격 —
primary/sweep/second_pass 3모드 공용).

- **읽기 범위 ≠ 보고 범위 (핵심 규칙).** 읽기는 저장소 전체 자유다 — 상류 검증·호출
  계약 확인으로 오탐을 줄이고, 그룹 밖에서 시작·종료하는 흐름을 끝까지 추적해 미탐도
  줄인다. 보고는 결함 라인이 자기 그룹 안에 있는 것만(소유자 유일 → 중복 보고 없음,
  기계 검증). 타 그룹 결함 징후는 `cross_refs` 힌트로 남긴다 — Stage 2.5가 반드시
  소비한다.
- **coverage — 파일별 정독 증거.** 그룹의 모든 core 파일에 경로 + 역할 + 최상위 위험
  가설(없으면 "특이점 없음" + 근거)을 기록해야 한다. 형식만 유효한 부분 커버리지
  (조용한 미탐)를 스크립트가 그룹 파일 목록과 대조해 잡아낸다.
- **프롬프트 인젝션 방어.** 저장소 내용은 신뢰할 수 없는 분석 대상 데이터다. "검토
  완료됨" 류 지시문은 따르지 않고 의심 신호로 격상한다. 하네스가 자동 주입하는 대상
  저장소의 CLAUDE.md·AGENTS.md도 동일하게 무력화한다(에이전트 본문에 명시). 감사 대상
  코드의 **실행은 금지**다 — 인젝션의 실행 채널이자 로컬 환경 차이로 인한 오판
  채널이다.

### Stage 2.5 — 보강 패스

`cross_refs`를 기록만 하고 소비하는 경로가 없으면 힌트가 파일에서 사장된다 — 이를
메우는 단계다. 순서가 의미 있다: ① `high_risk` 그룹에 **critical 전용 2차 헌터**를
먼저 투입·병합해(2차 헌터의 힌트까지 라우팅 대상에 포함) → ② `route-hints`가 전체
힌트를 소유 그룹으로 라우팅하고(기존 발견이 커버하는 힌트는 제외 — 위치 중첩 + 동일
category 기준) 남은 힌트 그룹에 **표적 sweep 헌터**를 투입 → ③ sweep 라운드는 1회
고정(무한 재귀 방지)이며, 그 후에도 남은 힌트는 `--residue-check`가 걸러 보고서에
"미소진 힌트"로 표면화한다 — **포기하되 은폐하지 않는다.**

2차·sweep 산출은 기존 결과를 열람하지 않고 **별도 파일에 독립 기록**하며, 병합은
스크립트(`validate_output.py merge`)만 수행한다 — LLM의 읽기-수정-재기록은 기존
발견을 소실시켜도 스키마 검증을 통과하는 무성 소실 채널이라 금지다. 병합은 중복 제거
+ ID 유일성 + **기존 발견 ID 상위집합 검사**(병합 전 발견 전부 보존 확인)를 거친다.

### Stage 3 — 적대적 검증

발견이 있는 그룹마다 **새 컨텍스트** 검증자를 위임한다(`subagent_type:
deep-audit-verifier`, 헌터와 컨텍스트 비공유).

**독립성 프로토콜(anti-anchoring).** 검증자가 헌터 논리에 편승하는 확증 편향을
차단한다: ① 스크립트가 rationale을 제거한 발췌(`claims/<gid>.json` —
id·위치·claim·심각도만)를 만들어 임베드하고, ② 검증자는 claim만 보고 코드에서 결함을
**독립 재도출**해 `rederivation`에 기록하며, ③ **모든 발견의 재도출을 마친 뒤에만**
헌터의 파일을 열어 대조·채점한다. ④ 편승 징후(재도출이 헌터 rationale과 자구 유사)가
보이면 2턴 분리로 격상한다.

**룰브릭 — 3상태(met/unmet/unknown) 5기준.** ① 코드가 실제로 그러함 ② 전제조건 도달
가능 ③ 결과가 실재 악영향 ④ 상류 방어 없음 ⑤ 적대적 반박 생존. confirmed 판정
3규칙: **게이트**(unmet 1건 확정 → 점수 무관 배제), **임계**(met 3개 미만 → 배제;
5가 아니라 3인 이유는 unknown 허용 — 런타임 의존으로 확인 불가한 기준이 남아도
통과시키되 감정 보강 대상), **시나리오**(confirmed에는 구체적 실패 시나리오 필수).
boolean이 아니라 3상태인 이유: "거짓으로 확정"과 "확인 불가"를 구분하지 못하면 상류
방어를 실제 확인했는데도 점수상 confirmed가 성립하는 구멍이 생긴다. 판정 일관성은
`validate_output.py`가 **기계 검증**한다. met/unmet 판정에는 증거(file:line)가
의무이고, 임계 기각 전에는 unknown 해소 시도가 의무다(기각 오판 = 미탐).

**심각도 차등(원칙 4).** critical/major는 5기준 전체 + **감정 보강**(unknown 잔존 시
호출부·설정·스레드 모델 추가 추적, 이력을 `appraisal`에 기록); minor는 ①③ 경량
확인 + 명백한 상류 방어 1회 확인. 검증자는 심각도 재조정 권한을 갖되 **minor 격상
시 풀 룰브릭 재채점 의무**다. confirmed에는 개선 방향과 코드 샘플을 작성한다.

### Stage 4 — 보고서 생성

`build_report.py`가 verified+defects를 조인해 **한국어** 보고서를 렌더링한다. 기준
독자는 **코드베이스에 낯선 검토자** — 발견별로 claim 제목, 위치·심각도·검증 점수,
파일 역할, 비전문 언어의 영향 한 줄, 결함 스닙, 도달 경로, 메커니즘, 실패 시나리오,
접이식 검증 노트(반박·가드 스캔·감정 이력), 개선 샘플을 싣는다. 요약부에는 용어
범례·발견 색인·감사 범위·**검증 불능 그룹**(감사 공백 — 수동 확인 권장)을 명시한다.
15건 초과 시 3분할(`00_요약.md` / `01_critical_major.md` / `02_minor.md`). 서식
명세는 [references/report-format.md](../../skills/deep-code-audit/references/report-format.md).

### 오케스트레이션 정책

- **동시성**: 오케스트레이터가 직접 관리 — 상한(4~6)만큼 백그라운드 스폰, 완료
  통지마다 다음 그룹 스폰.
- **재시도 — 실패 양식 3종 구분**: 스키마 불합격(파일 있음) → 동일 에이전트에 오류
  회신 1회; 무응답·크래시(파일 없음) → 새 스폰 1회; 재실패 → 헌터는 이분할, 검증자는
  묶음 축소 재시도, 그래도 실패하면 `failed` 기록 후 보고서에 "검증 불능 그룹" 명시.
- **호출 인터페이스**: 자유 텍스트 — 대상 루트·라인 예산·include/exclude·재개를
  플래그와 자연어("tests 빼고", "아까 하던 감사 이어서") 양쪽으로 해석한다.
- **언어 정책**: 모델에게 주는 지시(SKILL.md·에이전트 본문·태스크 골격·스키마 산문)는
  **영어**(토큰 비용 — 지시층은 스폰 수 × 턴 수로 곱셈 과금, 전환으로 지시층 ~36%
  절감), 사람에게 닿는 텍스트(산출 JSON 산문 필드·보고서 렌더)는 **한국어**.
  `validate_output.py`가 고신호 산문 필드의 한글 전무를 경고하는 드리프트 감지기를
  겸한다.

## 7. 운영 파라미터와 남은 작업

| 항목 | 확정값 | 튜닝 신호 (M4) |
|------|--------|----------------|
| 그룹 라인 예산 | 10,000 (CLI 조정 가능) | cross-file FN → 상향 / 헌터 품질 저하 → 하향 |
| 검증 임계 | met 3 | FP 과다 → 4 / critical FN → critical 한정 2 검토 |
| 2차 패스 범위 | high_risk 그룹만 | 비고위험 critical FN → 고위험 판정 기준 확장 |
| 동시 실행 상한 | 4~6 | 운영 환경 리소스 |
| 보고서 분할 | 15건 초과 시 3분할 | — |

**남은 마일스톤은 M3·M4**다:

- **M3 — 평가 fixture + 채점 스크립트**: 결함 ~20건을 심은 소형 저장소 + 정답지 +
  TP/FP/FN 자동 집계. fixture는 미탐 취약 지점을 겨냥한다 — cross-file 결함(의도적
  타 그룹 배치), 오탐 미끼(상류 방어 있는 코드), 프롬프트 인젝션 미끼. 라인 예산을
  하향(예: 2,000)해 최소 3그룹을 강제한다 — 그룹 경계·힌트 라우팅·sweep이라는 미탐
  방어 기제가 실행되게 하기 위해서다.
- **M4 — 측정·튜닝**: precision/recall 측정(critical/major recall이 핵심 지표,
  3회 실행 분산 기록) + 결함 없는 실환경 저장소의 오탐률 별도 측정 + 매 실행
  `issues.jsonl` 회고로 프롬프트·스크립트 수정 + `description` 발동 정확도 측정.

## 8. 설계 이력 (통합·대체된 선행 문서 요약)

이 문서는 다음 선행 문서를 모두 통합·대체한다. 개별 설계 근거는 본문 §2~§5에 흡수됐다.

1. **워크플로우 정합성 검토** (초기): 4단계 파이프라인·파일 인계·읽기/보고 분리·
   3단계 심각도·보고서 분할의 원형 정의.
2. **개발 계획 (Fable 최적화 확정판, 2026-07-02)**: 현행 설계 대부분 확정 — 판단의
   인모델 이관(스크립트 6종→4종), Stage 2.5 신설, 3상태 룰브릭·게이트,
   anti-anchoring, coverage 증거, seam_hints, 자기 배제, state.json, 재시도 정책.
   5차 설계 리뷰로 결함 30여 건 사전 보완.
3. **파서 개선 (2026-07-03)**: 첫 실전 감사(ssam — 100% Rust)에서 import 파서 부재로
   응집 그룹핑이 **조용히** 무력화된 사건이 계기. Rust/Go/Java·Kotlin 파서 추가 +
   저하 감지·고지(원칙 6 승격). 재검증: 간선 0→142.
4. **전용 서브에이전트 번들링 (2026-07-06)**: 프로토콜이 오케스트레이터 컨텍스트를
   경유하던 약점(컴팩션 의역 위험·템플릿 누적·타입 오선택)을 에이전트 본문 이관 +
   변수만 담은 task 골격으로 해소. Stage 0 프리플라이트 신설. 스키마 상대경로·자동
   주입 인젝션 표면 2건 동시 교정.
5. **지시층 영어 전환 (2026-07-07)**: 지시층은 영어(스폰 수 × 턴 수 곱셈 과금 —
   지시층 유효 입력 약 −36%, 런당 ~340K 토큰 절감 추정), 데이터층·렌더층은 한국어
   유지라는 언어 정책 불변식 확립. `rederivation` 한국어는 편승 검사의 전제 조건.
   잔여 한글 검사 허용 목록 + 고신호 필드 한글 전무 경고(드리프트 감지기) 동반.
6. **정탐 정밀도 개선 (2026-07-07 계획, Phase 1~3 구현)**: 브리프에 실행 환경 사실
   (OS·아키텍처·동시성·노출 모델) 추가로 unknown 양산 차단(트랙 A), 룰브릭 운영화
   (트랙 B) — met/unmet 판정에 증거 의무, 임계 기각 전 unknown 해소 시도 의무,
   검증자 본문에 심각도 앵커, 조건부 confirmed의 전제 표면화. 렌즈 우선순위의 배제적
   해석 금지, 대상 코드 실행 금지 명문화.
7. **플러그인 구조 전환 (2026-07-13)**: `.claude-plugin/plugin.json` 신설, 스킬
   본체를 `skills/deep-code-audit/`로 이동, `agents/`는 플러그인 표준 위치(루트)
   유지. 설치가 심링크 1개로 축소. 에이전트가 스코프명으로 등록되므로 Stage 0a에
   식별자 확정 절차 추가. 레거시 에이전트 심링크는 전환기 안전망으로 병행.
