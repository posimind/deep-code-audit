# 전용 서브에이전트 번들링 구현 플랜

deep-code-audit의 헌터·검증자를 **전용 서브에이전트 정의**(`agents/*.md`)로 승격해,
프로토콜 전달 충실도와 오케스트레이터 컨텍스트 효율을 개선하는 작업 계획.
2026-07-04 검토 결과를 근거로 하며, 구현 전 사실관계는 부록 A에 정리했다.

## 0. 배경과 목표

### 왜 하는가 (현재 설계의 약점 4가지)

| # | 약점 | 에이전트화가 주는 해법 |
|---|------|------|
| 1 | 불변 프로토콜(비대칭 기록·읽기/보고 분리·인젝션 방어·룰브릭)이 **오케스트레이터 컨텍스트를 경유**해 전달됨 — 긴 런에서 컴팩션 후 의역·축약 위험(조용한 품질 저하의 프롬프트판) | 프로토콜을 에이전트 본문(=시스템 프롬프트)으로 옮기면 하네스가 **디스크에서 직접 로드** — 바이트 안정성 확보 |
| 2 | 스폰마다 템플릿 전문(~130줄)이 Agent 호출 파라미터로 오케스트레이터 컨텍스트에 누적 (그룹 10개면 수십 회 반복) | 스폰 프롬프트가 **변수 몇 개로 축소** → 컴팩션 빈도 감소 → 런 후반부 오케스트레이션 오류 위험 감소 |
| 3 | "읽기 전용 탐색 타입 금지" 같은 **자연어 지시로 에이전트 타입 선택** — 잘못 고르면 산출 JSON을 못 씀 | `subagent_type` 명시 지정으로 결정화 |
| 4 | 헌터·검증자가 전체 도구를 상속 — 감사 대상 저장소를 **수정할 수 있고**, 인젝션 시 blast radius가 큼 | `tools` allowlist로 Edit/MCP 차단 (단 Bash가 있는 한 보안 경계가 아닌 **사고 방지** 수준 — 부록 A 참고) |

### 비목표

- 오케스트레이터 자체의 서브에이전트화(`context: fork`) — 사용자 확인
  (Stage 1d 응집 저하 고지 등)에 `AskUserQuestion`이 필요한데 서브에이전트에서는 불가.
- 모델·effort 튜닝 — 프런트매터에 자리만 확보하고 값은 M4 측정에서 결정.
  현행 설계 결정("모델 미지정·상속")을 유지한다(프런트매터 `model` 생략 = inherit).
- 스크립트 4종 변경 — 이번 작업은 프롬프트 전달 경로만 바꾸며 스크립트는 무변경.

## 1. 채택 설계

### 1a. 에이전트 2종

| | `deep-audit-hunter` | `deep-audit-verifier` |
|---|---|---|
| 역할 | Stage 2 primary + Stage 2.5 sweep/second_pass (3모드 겸용, 모드는 태스크 프롬프트로 선택) | Stage 3 적대적 검증 (단일/batch/2턴 분리 공용) |
| 본문(시스템 프롬프트) | 현 `hunt-agent.md`의 불변 프로토콜 전부 | 현 `verify-agent.md`의 불변 프로토콜 전부 |
| `tools` | `Read, Grep, Glob, Bash, Write` | 동일 |
| `model` / `effort` | 생략(inherit) — M4 튜닝 노브 | 동일 |
| `maxTurns` | v1 미설정 (10K라인 그룹 정독에 필요한 턴 수 데이터가 없음 — M4에서 측정 후 설정) | 동일 |
| `description` | "deep-code-audit 오케스트레이터가 명시 지정할 때만 사용, **다른 작업에 자동 위임 금지**" 명문화 | 동일 |

sweep/second_pass를 별도 에이전트로 만들지 않는 이유: 세 모드는 현 골격에서도 "모드
스위치" 절만 다르고, 에이전트를 늘리면 프로토콜 본문이 3중 복제되어 drift 채널이 생긴다.

### 1b. 배치 방식 — B안 기본, A안은 배포 시점 옵션

- **B안 (기본)**: 레포에 `agents/` 디렉터리 신설 + 심링크
  `~/.claude/agents/deep-code-audit → <repo>/agents`.
  - 사용자 스코프 `~/.claude/agents/`는 **재귀 스캔**되고 하위 경로는 식별자에 영향
    없음(식별자 = frontmatter `name`). 파일 감시로 라이브 반영(재시작 불요).
  - 스킬의 호출명·트리거 description이 **전혀 변하지 않음**.
- **A안 (보류)**: 스킬 폴더에 `.claude-plugin/plugin.json` 추가 → skills-dir 플러그인화.
  - 공식 번들링 경로지만, 플러그인 루트 SKILL.md는 **호출명이 플러그인 스코프로 바뀌고**
    (`/deep-code-audit` → `/<플러그인명>:<name>` 형태), 에이전트 ID도 스코프化되며,
    `agents/` 변경에 `/reload-plugins`가 필요해 개발 루프가 무거워진다.
  - 타인 배포가 필요해지는 시점에 재평가한다. B안의 레포 구조(`agents/` 디렉터리)는
    A안 전환 시 그대로 재사용된다.

> 이전 검토에서는 A안을 우선 권고했으나, 플러그인화가 스킬 호출명을 바꾼다는 점
> (정교하게 튜닝된 트리거 identity 훼손)과 현 단일 사용자 환경을 반영해 B안을 기본으로
> 조정했다. Phase 0 스파이크에서 B안 동작을 확정한다.

### 1c. 템플릿 분리 원칙 — 단일 소스 유지

```
references/hunt-agent.md   ──분리──▶  agents/deep-audit-hunter.md   (불변 프로토콜 = 본문)
                                      references/hunt-task.md       (런 변수 골격 = 스폰 프롬프트)
references/verify-agent.md ──분리──▶  agents/deep-audit-verifier.md
                                      references/verify-task.md
```

- 분리 후 **원본 2파일은 삭제**한다. 남겨두면 이중 소스 drift 채널이 된다.
- **호환 모드도 단일 소스로**: 전용 타입을 쓸 수 없는 환경에서 사용자가 **명시
  동의한 경우에만**(§1e), 오케스트레이터가 `agents/<name>.md` **본문** +
  `references/<x>-task.md`를 이어붙여 기존 방식(범용 서브에이전트 + 전체 프롬프트)으로
  전달한다. 별도 통합본은 유지하지 않는다. 단 이 모드는 텍스트만 같을 뿐 전달 경로
  약점(§0 #1·#2)이 되살아나고 도구 제한(#4)도 사라지므로 **자동 발동은 금지** —
  발동 조건과 중단 정책은 §1e가 규정한다.

### 1d. 태스크 프롬프트(스폰 프롬프트) 변수 목록

| 변수 | hunter | verifier | 비고 |
|------|:---:|:---:|------|
| `{{TARGET_ROOT}}` | ✓ | – | |
| `{{RUN_DIR}}` | ✓ | ✓ | |
| `{{GROUP_ID}}` | ✓ | ✓ | |
| `{{OUTPUT_PATH}}` | ✓ | ✓ | 모드/batch에 따라 상이 |
| `{{MODE_INPUTS}}` + 모드 스위치 | ✓ | – | primary/sweep/second_pass |
| `{{CLAIMS_EMBED}}` | – | ✓ | rationale 미포함 원칙 유지 |
| `{{SCHEMA_PATH}}` | ✓ | ✓ | **신규** — 아래 잠재 결함 교정 |

**잠재 결함 교정 (이번 작업에 포함)**: 현 골격은 산출 스키마를 `references/schemas.md §2`
상대경로로 지시하는데, 서브에이전트의 CWD는 **감사 대상 루트**라 이 경로가 해석되지
않는다(오케스트레이터가 알아서 절대경로로 바꿔줬기를 기대하는 암묵 의존). 에이전트 본문은
치환이 불가능하므로, 스키마 위치는 태스크 프롬프트의 `{{SCHEMA_PATH}}`(= `$SKILL/references/schemas.md`
절대경로)로 명시 전달한다.

### 1e. 프리플라이트와 중단 정책 (2026-07-04 후속 검토 반영)

지침을 에이전트 정의로 옮기면 **에이전트 인식 실패 = 프로토콜 미전달**이다. 이를 모른 채
진행하는 것은 이 스킬이 크래시보다 위험하게 취급하는 실패 양식(조용한 품질 저하)이므로,
초안의 "자동 폴백"은 철회하고 아래 정책으로 대체한다.

**왜 기본값이 중단인가**: 호환 모드는 §1c에 적었듯 본선 동등이 아니다(전달 경로 약점
#1·#2 부활, 도구 제한 #4 상실). 거의 실행되지 않는 경로가 감지 없이 자동 발동하는 구조
자체가 silent-degradation 채널이다. 반면 state.json 재개 아키텍처 덕에 **중단 비용은
낮다** — 환경을 고치고 재개하면 완료분은 건너뛴다. 중단이 싸고 저품질 완주가 비싸므로
기본값은 중단이어야 한다.

**정책 6항**:

1. **Stage 0 프리플라이트** (신규 런·재개 공통, Stage 1 진입 전):
   - (a) **정의 파일 확인** (결정적): 두 에이전트 파일이 설치 위치에 존재하고 frontmatter
     `name` 이 기대값과 일치하는지 ls/grep 으로 확인.
   - (b) **세션 인식 확인** (인모델): 사용 가능한 subagent 타입 목록에 두 타입이 보이는지
     확인. 파일은 있는데 안 보이면 세션 시작 전 `~/.claude/agents/` 부재가 원인일 수
     있음(문서화된 동작) → 세션 재시작 안내.
   - (c) **카나리 스폰** (동작 확인): 두 에이전트를 각각 최소 과제("`$RUN/preflight/<name>.json`
     에 ok 기록")로 동시 스폰 후 파일 확인 — 스폰 가능성·Write 동작·run 디렉터리 쓰기
     권한을 한 번에 검증한다. 비용은 미니 스폰 2회로 전체 런 대비 무시 가능. 남은 미완료
     단계가 서브에이전트를 요구하지 않으면(보고서만 남은 재개) 생략 가능.
   - **순서**: (a)·(b)는 **run 디렉터리 생성·확정 전에** 수행한다 — 여기서 실패하면
     잔여물 없이 중단된다. (c)는 run 디렉터리 확정 후 수행한다.
2. **실패 시 기본 중단**: 어느 검사가 왜 실패했는지 + 교정 방법(심링크 설치 명령, 세션
   재시작)을 보고하고 **Stage 1 로 진행하지 않는다**. run 디렉터리가 이미 있으면 실패를
   `log-issue --stage preflight` 로 기록한다(없으면 대화 보고만 — 기록 위치가 없다).
3. **호환 모드는 명시 동의로만**: 저하 내용(프로토콜 전달 경로·도구 제한 상실)을 고지받은
   사용자가 명시적으로 동의한 경우에만 §1c 결합 방식으로 진행한다. 동의 사실을
   `log-issue` 로 기록하고, 최종 사용자 보고에 호환 모드 실행임을 명시한다. 호환 모드
   태스크 프롬프트에는 **Edit 등 대상 파일 수정 금지**를 명시해 frontmatter 도구 제한
   상실을 프롬프트 수준에서나마 보완한다. Stage 1d(응집 저하 고지 후 진행 여부 확인)와
   동일한 패턴 — 진행하되 동의 없이 저하하지 않는다.
4. **런 중간 인식 상실 = 인프라 실패**: 런 도중 스폰이 타입 미인식으로 실패하면(재개
   세션의 환경 변화 등) 해당 그룹을 범용 타입으로 강등하지 않는다 — 혼합 모드는 그룹 간
   결과 비교 가능성을 깨고 문제를 가린다. `log-issue` 후 중단하고 환경 교정 → 재개를
   안내한다. 산출물 불량에 대한 기존 재시도 사다리(후속 회신 → 재스폰 → 이분할/축소)와는
   별개 실패 계열이며, 그쪽은 불변이다.
5. **모드 기록과 재개 일관성**: 프리플라이트 통과(또는 호환 동의) 시
   `$RUN/preflight/mode.json` 에 `{"mode": "dedicated" | "compat"}` 를 기록한다
   (오케스트레이터가 직접 쓰는 비스키마 정보 파일 — 스크립트 무변경 원칙 유지).
   **재개 시 기록된 모드와 현재 환경이 다르면**(compat 런인데 전용 타입이 생겼거나,
   dedicated 런인데 사라짐) 자동 전환하지 않고 차이를 고지한 뒤 사용자 결정을 받는다 —
   그룹 간 처리 방식이 갈리는 혼합 런은 4항과 같은 이유로 동의 없이는 금지.
6. **잔여 디렉터리 규칙**: 카나리 실패로 중단된 run 디렉터리에는 state.json 이 없다
   (init-state 는 Stage 1e). 재호출 시 최신 run 디렉터리에 state.json 이 없으면
   "프리플라이트에서 중단된 런"으로 간주해 **그 디렉터리에서 프리플라이트부터 재시도**
   한다 — 실패할 때마다 새 run 디렉터리가 늘어나는 것을 막는다. (현행 SKILL.md Stage 0
   의 재개 판별은 state.json 존재를 전제하므로 이 규칙을 명시하지 않으면 판별이 모호해진다.)

## 2. 단계별 작업

### Phase 0 — 타당성 스파이크 (반나절, 코드 변경 없음)

임시 에이전트 파일 1개로 아래를 확인한다. **하나라도 실패하면 Phase 1 진입 전에 설계 재조정.**

- [ ] `~/.claude/agents/deep-code-audit/`(심링크) 하위 에이전트가 세션에서 인식되는가
      (`@agent-` 타입어헤드 또는 Agent 도구 subagent_type으로 확인)
- [ ] 심링크 경유 파일 수정이 라이브 반영되는가 (스킬 심링크와 동일 동작 기대)
- [ ] `tools: Read, Grep, Glob, Bash, Write` 에이전트에서 Edit 호출이 실제 차단되는가
- [ ] 스폰 시 태스크 프롬프트가 본문(시스템 프롬프트)과 별개로 전달되고, 본문이
      오케스트레이터 컨텍스트에 나타나지 않는가
- [ ] 감사 대상 레포(스킬 레포가 아닌 곳)에서 시작한 세션에서도 에이전트가 보이는가
      (사용자 스코프이므로 보여야 정상)
- [ ] 재시도 정책 1번(동일 에이전트 후속 메시지 회신)이 전용 타입에서도 동작하는가
- [ ] 존재하지 않는 subagent_type 지정 시 실패 양상 확인(오류 메시지·동작) —
      §1e-4(런 중간 인식 상실)의 감지 시그니처 확보
- [ ] 전용 에이전트 컨텍스트에 **대상 레포의 CLAUDE.md 가 주입되는지 실측** — 공식
      문서상 커스텀 서브에이전트 경로는 CLAUDE.md 를 로드한다(부록 A). 주입된다면
      Phase 1 의 방어 문구 보강이 필수가 된다

## 부록 A — 공식 문서 사실관계 (2026-07-04 확인, code.claude.com/docs)

- **에이전트 정의 위치와 우선순위**: managed > `--agents` CLI > `.claude/agents/`(프로젝트)
  > `~/.claude/agents/`(사용자) > 플러그인 `agents/`. 사용자·프로젝트 스코프는 재귀
  스캔되며 하위 디렉터리는 식별자에 영향 없음(식별자 = frontmatter `name`). 파일 감시로
  라이브 반영.
- **frontmatter 지원 필드**: `name`(필수), `description`(필수), `tools`,
  `disallowedTools`, `model`(기본 inherit), `permissionMode`, `maxTurns`, `skills`(프리로드),
  `mcpServers`, `hooks`, `memory`, `background`, `effort`, `isolation`, `color`,
  `initialPrompt`. 본문 마크다운 = 시스템 프롬프트(서브에이전트는 메인 시스템 프롬프트를
  받지 않고 본문 + 기본 환경 정보만 받는다).
- **플러그인 에이전트 제약**: `hooks`·`mcpServers`·`permissionMode`는 보안상 무시됨
  (이 스킬에는 셋 다 불필요).
- **skills-dir 플러그인**: 스킬 폴더에 `.claude-plugin/plugin.json` 추가 시
  `<name>@skills-dir` 플러그인으로 로드되어 `agents/` 번들 가능. 단 플러그인 루트
  SKILL.md의 호출명은 플러그인 스코프로 바뀌며, `agents/` 변경 반영에 `/reload-plugins`
  필요.
- **스킬 frontmatter `context: fork` + `agent:`**: 스킬 전체를 지정 에이전트 타입으로
  실행하는 기능. 서브에이전트에서는 `AskUserQuestion`·`EnterPlanMode` 등 UI 의존 도구
  사용 불가 → 사용자 확인이 필요한 이 스킬의 오케스트레이터에는 부적합.
- **서브에이전트 CWD**: 메인 세션의 작업 디렉터리에서 시작(감사 시나리오에서는 대상
  루트). `cd`는 Bash 호출 간 비지속. → 상대경로 의존 금지, §1d의 SCHEMA_PATH 교정 근거.
- **CLAUDE.md 주입**: 문서상 커스텀 서브에이전트 실행 경로에는 CLAUDE.md 가 로드된다
  (내장 Explore/Plan 은 CLAUDE.md·git status 를 건너뜀). 감사 시나리오에서는 **대상
  레포의 CLAUDE.md** 가 헌터·검증자에게 특권 채널로 주입될 수 있다 → Phase 0 실측 +
  Phase 1 본문 방어 문구의 근거.
- **도구 제한의 한계**: `tools` allowlist는 Edit/MCP를 막지만 Bash가 포함되는 한 임의
  파일 조작이 원리상 가능 — 도구 제한은 사고 방지책이며 보안 경계는 세션 권한 설정의 몫.
